Сценарий

Сортировка вставками

Мы решаем задачу сортировки: нам нужно упорядочить ключи K1,K2,…,KnK_1, K_2, \dotsc, K_nK1​,K2​,…,Kn​ по возрастанию относительного какого-то линейного порядка.

Метод простых вставок

Допустим, мы уже отсортировали j−1j-1j−1 записей так, что

K1⩽K2⩽⋯⩽Kj−2   ⁣⩽   ⁣Kj−1K_1 \le K_2 \le \dotsb \le K_{j-2} \;\! \le \;\! K_{j-1}K1​⩽K2​⩽⋯⩽Kj−2​⩽Kj−1​

Теперь нам нужно вставить следующий ключ KjK_jKj​. Будем сравнивать его по очереди с ключами Kj−1,Kj−2,…K1K_{j-1}, K_{j-2}, \dotsc K_1Kj−1​,Kj−2​,…K1​, пока не обнаружим, что его нужно вставить между ключами KiK_iKi​ и Ki+1K_{i+1}Ki+1​. Сдвинем все ключи, начиная с Ki+1K_{i+1}Ki+1​, на одну позицию и вставим запись KjK_jKj​ на место i+1i+1i+1.

Другими словами, бы каждый раз берем новый ключ и вставляем его в правильное место.

function insertion_sort(mutable array a[n]): for int j = 1; j < n; j++: int i = j - 1 key = a[j] while i >= 0 and a[i] > key: a[i+1] = a[i] i -= 1 a[i+1] = key

Количество сравнений в этом алгоритме равно числу инверсий перестановки (K1,K2,…,Kn)(K_1, K_2, \dotsc, K_n)(K1​,K2​,…,Kn​) плюс количество выходных сравнений в цикле while — n−1n-1n−1

min⁡=n−1,ave⁡=(n2+3n−4)/4,max⁡=(n2+n−2)/2,dev⁡=16n (n−1) (n+5/2)\min\limits = n-1 ,\quad \ave = (n^2 + 3n - 4) / 4 ,\quad \max\limits = (n^2 + n - 2) / 2 ,\quad \dev = \frac{1}{6}\sqrt{n \, (n-1) \, (n + 5/2)} min=n−1,ave=(n2+3n−4)/4,max=(n2+n−2)/2,dev=61​n(n−1)(n+5/2)​

Количество записей (присваиваний элементов a) совпадает с количеством сравнений.

Бинарные и двухпутевые вставки

В алгоритме простых вставок новый jjj-й элемент сравнивался в среднем с (j−1)/2(j-1)/2(j−1)/2 уже отсортированными элементами по порядку. В итоге общее количество сравнений в среднем равно ∑j=2n(j−1)/2=(n2−n)/4=O(n2)\sum\limits_{j=2}^n (j-1)/2 = (n^2 - n) / 4 = O(n^2)j=2∑n​(j−1)/2=(n2−n)/4=O(n2).

Можно использовать то, что элементы, в которые мы хотим вставить новый ключ, уже отсортированы. Будем искать место вставки с помощью бинарного поиска. До конца это проблему не решит, ведь кроме поиска места вставки нужно её произвести. Но даже так это неплохой буст к производительности

Итого мы для каждого элемента с номером j+1j+1j+1 проводим бинарный поиск в подмассиве размера jjj (все, что находится спереди). В итоге максимальное количество сравнений

∑j=1n−1(⌊log⁡2j⌋+1)=n⋅⌈log⁡2n⌉−2⌈log⁡2n⌉+1=nlog⁡2n−n+O(1)\sum\limits_{j=1}^{n-1} \bigl( \lfloor \log_2 j \rfloor + 1 \bigr) = n \cdot \lceil \log_2 n \rceil - 2^{\lceil \log_2 n \rceil} + 1 = n \log_2 n - n + O(1)j=1∑n−1​(⌊log2​j⌋+1)=n⋅⌈log2​n⌉−2⌈log2​n⌉+1=nlog2​n−n+O(1)

Мы получили, что количество сравнений близко к оптимальному, и это не может не радовать. Однако, наша сортировка всё равно выполняет O(n2)O \bigr( n^2 \bigr)O(n2) записей, что сводит на нет все наши старания.

Проблема в том, что вставка в массив выполняется за O(n)O(n)O(n) записей. Вообще, не существует линейной структуры данных, обеспечивающей одновременно поиск и вставку быстрее, чем за O(n)O(n)O(n) операций. Можно использовать нелинейные структуры, например сбалансированные деревья поиска или список с пропусками, но тогда у нас будет нехилый временной оверхед на саму структуру.

Смягчить проблему можно, рассмотрев особый метод двухпутевых вставок.

Двухпутевые вставки

Выделим под массив участок памяти размером 3n3n3n, а сам массив поместим в середину. Будем поддерживать указатели на начало массива и его длину, чтобы не потерять массив в аллоцированном участке.

При вставке элемента в массив определяем, к какому краю ближе место вставки jjj, и расширяем массив в сторону этого края, копируя min⁡(n−j,j)\min\limits(n-j, j)min(n−j,j) элементов вместо n−jn-jn−j. Таким образом мы в 222 раза уменьшили ожидаемое количество записей, получив алгоритм со средним количество записей n2/8+O(n)n^2/8 + O(n)n2/8+O(n) вместо n2/4+O(n)n^2/4 + O(n)n2/4+O(n).

Блочные вставки

Есть способ немного сгладить проблему линейных структур данных. Нужно использовать блочный список.

Массив представляем в виде блочного списка с размером блока kkk. Поиск в блочном списке длины nnn занимает O(n/k+log⁡k)O(n/k + \log k)O(n/k+logk) операций, а вставка занимает O(k)O(k)O(k) операций, если мы знаем, в какой блок вставлять (имеем на него ссылку).

Получается, что общее время работы алгоритма сортировки блочными вставками равно

∑j=1n(O(j/k+log⁡k)+O(k))=O(n2k+nlog⁡k+nk)\sum\limits_{j=1}^n \bigl( O(j/k + \log k) + O(k) \bigr) = O \left( \frac{n^2}{k} + n \log k + nk \right)j=1∑n​(O(j/k+logk)+O(k))=O(kn2​+nlogk+nk)

Минимум достигается при k=nk = \sqrt{n}k=n​, общее количество операций при сортировке блочными вставками получается равным O(n3/2)O \bigl( n^{3/2} \bigr)O(n3/2).

Метод экзотический, имеет какой-то оверхед на структуру и не является широко универсальным, так как приходится следить за размерами блоков, совершать реаллокации и слияния.

Такой механизм предложил Дональд Льюис Шелл в 1959 году. На основе этого механизма построена сортировка Шелла.

Основная идея — разбивать элементы на пары так, чтобы расстояние между элементами в одной паре было равно степени двойки.

input mutable array[int] a[n] input array[int] h[t] for gap in h : for int j = gap; j < n; j++: current = a[j] i = j - gap while i >= 0 and current < a[i]: a[i + gap] = a[i] i -= gap a[i + gap] = current
