Сценарий

Компактные структуры данных

Пусть zzz — минимальное с точки зрения теории информации количество бит, требуемое для хранения информации о данных. Компактная структура данных — структура, хранящая все эти данные, и при этом использующая максимально близкое к zzz количество бит.

Компактные структуры данных еще называют безызбыточными.

Битовые массивы

Компактные индексируемые словари

Компактный индексируемый словарь или RSD (Rank-Select Dictionary) — битовый массив, поддерживающий две дополнительные операции rank и select.

Сам битовый массив — массив из nnn битов (здесь везде индексы, а не номера, поэтому счёт начинается с 000)

b0 b1 b2 ⋯ bn−2 bn−1b_0 \, b_1 \, b_2 \, \cdots \, b_{n-2} \, b_{n-1}b0​b1​b2​⋯bn−2​bn−1​

Rank — операция подсчёта битов на префиксе.

rank⁡1(k)\rank_1 (k)rank1​(k) — количество единичных битов на префиксе длины kkk, то есть на позициях от 000 до k−1k-1k−1:

rank⁡1(k)   ⁣=def   ⁣∑j=0k−1bj\rank_1 (k) \defeq \sum\limits_{j=0}^{k-1} b_jrank1​(k)=defj=0∑k−1​bj​

rank⁡0(k)\rank_0 (k)rank0​(k) — количество нулевых битов на префиксе длины kkk, то есть на позициях от 000 до k−1k-1k−1:

rank⁡0(k)   ⁣=def   ⁣∑j=0k−1(1−bj)=k−∑j=0k−1bj=k−rank⁡1(k)\rank_0 (k) \defeq \sum\limits_{j=0}^{k-1} (1 - b_j) = k - \sum\limits_{j=0}^{k-1} b_j = k - \rank_1 (k)rank0​(k)=defj=0∑k−1​(1−bj​)=k−j=0∑k−1​bj​=k−rank1​(k)

В операции rank спрятался веса Хемминга ν\nuν (количество единичных бит в числе, popcount). А именно,

ν((b0 b1 b2 ⋯ bn−2 bn−1)2)=rank⁡1(n)\nu \Bigl( \bigl( b_0 \, b_1 \, b_2 \, \dotsm \, b_{n-2} \, b_{n-1} \bigr)_2 \Bigr) = \rank_1 (n)ν((b0​b1​b2​⋯bn−2​bn−1​)2​)=rank1​(n)

Select — операция поиска позиции бита.

select⁡1(k)\select_1 (k)select1​(k) — позиция kkk-го единичного бита

select⁡1(k)   ⁣=def   ⁣min⁡{j:rank⁡1(j+1)=k}\select_1 (k) \defeq \min\limits \bigl\{ j : \rank_1 (j+1) = k \bigr\}select1​(k)=defmin{j:rank1​(j+1)=k}

select⁡0(k)\select_0 (k)select0​(k) — позиция kkk-го нулевого бита

select⁡0(k)   ⁣=def   ⁣min⁡{j:rank⁡0(j+1)=k}\select_0 (k) \defeq \min\limits \bigl\{ j : \rank_0 (j+1) = k \bigr\}select0​(k)=defmin{j:rank0​(j+1)=k}

Наивная реализация

Rank можно считать с помощью префиксных сумм. Для этого нужно завести отдельный массив RRR длины nnn и хранить в нём по номеру kkk сумму на префиксе длиной kkk:

Rk={0 if k=0∑j=0k−1bj otherwise R_k = \cases{0 & \if k = 0 \\ \sum\limits_{j=0}^{k-1} b_j & \otherwise}Rk​=⎩⎨⎧​0j=0∑k−1​bj​​ if k=0 otherwise ​

Теперь операцию rank можно выполнить за O(1)O(1)O(1), а точнее за одно обращение к массиву RRR:

rank⁡1(k)=Rk\rank_1 (k) = R_krank1​(k)=Rk​

Rank — неубывающая функция, и RRR — неубывающий массив. Значит, значение функции select можно найти бинарным поиском по значению rank.

Это вычисление сделает максимум ⌊log⁡2(n+1)⌋+1\lfloor \log_2 (n+1) \rfloor + 1⌊log2​(n+1)⌋+1 сравнений и займёт время O(log⁡n)O(\log n)O(logn).

Важное уточнение по поводу занимаемой массивом RRR памяти.

Исходные данные — битовый массив bbb, занимающий в памяти nnn бит. А в массиве RRR мы храним числа, и если это обычные 646464-битные числа, то занимаемая массивом RRR память составляет 64⋅(n+1)64 \cdot (n+1)64⋅(n+1) бит. Это слишком много.

Важное замечание, которое позволит сократить память. В массиве RRR мы храним количества единичных битов на префиксах массива bbb, а оно не может быть больше nnn, а значит размер хранимых чисел не превышает ⌊log⁡2n⌋+1\lfloor \log_2 n \rfloor + 1⌊log2​n⌋+1 бит.

То есть, например, если в битовом массиве bbb у нас 210−12^{10}-1210−1 бит, то для чисел массива RRR можно выделять по 101010 бит, а не по 646464.

В итоге занимаемая массивом RRR память оказывается (⌊log⁡2n⌋+1)⋅(n+1)\bigl( \lfloor \log_2 n \rfloor + 1 \bigr) \cdot (n + 1)(⌊log2​n⌋+1)⋅(n+1) бит.

Можно пойти дальше, и попытаться ещё оптимизировать память. R0=0R_0 = 0R0​=0, значит для хранения этого значения не нужно вообще места, R1⩽1R_1 \le 1R1​⩽1, значит для хранения этого значения нужно не больше 111 бита, R2⩽2R_2 \le 2R2​⩽2, значит для хранения этого значения нужно не больше 222 битов. Продолжая эти рассуждения, получаем, что для хранения RkR_kRk​ нужно не больше ⌊log⁡2k⌋+1\lfloor \log_2 k \rfloor + 1⌊log2​k⌋+1 бит. В итоге получаем, что минимальное количество памяти, требуемое для хранения массива RRR, равно

∑k=1n(⌊log⁡2k⌋+1)=∑d=1⌊log⁡2n⌋d⋅2d−1+(⌊log⁡2n⌋+1)⋅(n−2⌊log⁡2n⌋+1)=(⌊log⁡2n⌋+1)⋅(n+1)−2⌊log⁡2n⌋+1+1\sum\limits_{k=1}^n \bigl( \lfloor \log_2 k \rfloor + 1 \bigr) = \sum\limits_{d=1}^{\lfloor \log_2 n \rfloor} d \cdot 2^{d - 1} + \bigl( \lfloor \log_2 n \rfloor + 1\bigr) \cdot \bigl( n - 2^{\lfloor \log_2 n \rfloor} + 1 \bigr) = \bigl( \lfloor \log_2 n \rfloor + 1 \bigr) \cdot (n + 1) - 2^{\lfloor \log_2 n \rfloor + 1} + 1k=1∑n​(⌊log2​k⌋+1)=d=1∑⌊log2​n⌋​d⋅2d−1+(⌊log2​n⌋+1)⋅(n−2⌊log2​n⌋+1)=(⌊log2​n⌋+1)⋅(n+1)−2⌊log2​n⌋+1+1

Результат неплохой: для битового массива bbb размером 404040 МБ потребуется дополнительно примерно 111 ГБ на массив RRR.

Числа на позициях от 111 до jjj занимают ∑k=1j(⌊log⁡2k⌋+1)=(⌊log⁡2j⌋+1)⋅(j+1)−2⌊log⁡2j⌋+1+1\sum\limits_{k=1}^j \bigl( \lfloor \log_2 k \rfloor + 1 \bigr) = \bigl( \lfloor \log_2 j \rfloor + 1 \bigr) \cdot (j + 1) - 2^{\lfloor \log_2 j \rfloor + 1} + 1k=1∑j​(⌊log2​k⌋+1)=(⌊log2​j⌋+1)⋅(j+1)−2⌊log2​j⌋+1+1 бит. Значит, в таком способе хранения число jjj записано в битах на позициях с jm−2m+1j m - 2^m + 1jm−2m+1 по (j+1)⋅m−2m+1(j+1) \cdot m - 2^m + 1(j+1)⋅m−2m+1, где m=⌊log⁡2j⌋+1m = \lfloor \log_2 j \rfloor + 1m=⌊log2​j⌋+1 — количество бит в числе jjj.

Если эта последовательность бит находится в памяти с произвольным доступом, то время получения значения RjR_jRj​ почти константное: нам нужно вычислить начало отсчёта и выделить mmm бит.

Мы смогли реализовать битовый массив, поддерживающий операции rank за O(1)O(1)O(1) и select за O(log⁡n)O(\log n)O(logn), занимающий O(nlog⁡n)O(n \log n)O(nlogn) бит.

Декомпозиция

Наивная реализация довольно простая, и это, к сожалению, единственный её плюс. Всё ещё проблемой является количество дополнительной информации, требуемое для вычисления rank. Меньше бит на числа выделять не получится — мы и так взяли минимально возможный размер. Остаётся только хранить меньше чисел.

Давайте хранить в массиве RRR сумму не всех префиксов, а только префиксов чётной длины. То есть

Rk={0 if k=0∑j   ⁣=   ⁣02k−1bj otherwise для 0⩽k⩽⌊n/2⌋R_k = \cases{0 & \if k = 0 \\ \sum\limits_{j \;\! = \;\! 0}^{2 k - 1} b_j & \otherwise} \quad\text{для}~ 0 \le k \le \lfloor n/2 \rfloorRk​=⎩⎨⎧​0j=0∑2k−1​bj​​ if k=0 otherwise ​для 0⩽k⩽⌊n/2⌋
.

Тогда сам rank считается по формуле

rank⁡1(k)={Rm if k=2mRm+bk−1 if k=2m−1\rank_1 (k) = \cases{R_m & \if k = 2m \\ R_m + b_{k-1} & \if k = 2m - 1}rank1​(k)={Rm​Rm​+bk−1​​ if k=2m if k=2m−1​

То есть мы сократили потребление памяти в 222 раза, при этом сохранив константную временную сложность вычисления операции rank.

Развиваем идею. Будем хранить не каждый второй префикс, а каждый ℓ\ellℓ-й. Точнее, мы делим весь исходный массив bbb на блоки длиной ℓ\ellℓ, и в массив LLL будем сохранять только суммы префиксов до начала каждого блока. То есть

Lk={0 if k=0∑l   ⁣=   ⁣0k−1∑j=0ℓ−1blℓ+j=∑j=0kℓ−1bj otherwise для 0⩽k⩽⌊n/ℓ⌋L_k = \cases{0 & \if k = 0 \\ \sum\limits_{l \;\! = \;\! 0}^{k-1} \sum\limits_{j=0}^{\ell - 1} b_{l \ell + j} = \sum\limits_{j=0}^{k \ell - 1} b_j & \otherwise} \quad\text{для}~ 0 \le k \le \lfloor n / \ell \rfloorLk​=⎩⎨⎧​0l=0∑k−1​j=0∑ℓ−1​blℓ+j​=j=0∑kℓ−1​bj​​ if k=0 otherwise ​для 0⩽k⩽⌊n/ℓ⌋

Для вычисления rank нужно найти префикс и прибавить к нему сумму не более ℓ\ellℓ слагаемых

rank⁡1(k)=L⌊k/ℓ⌋+∑j=⌊k/ℓ⌋⋅ℓkbj\rank_1 (k) = L_{\lfloor k / \ell \rfloor} + \sum\limits_{j = \lfloor k / \ell \rfloor \cdot \ell}^k b_jrank1​(k)=L⌊k/ℓ⌋​+j=⌊k/ℓ⌋⋅ℓ∑k​bj​

Получается, что для вычисления операции rank мы совершим не более ℓ\ellℓ операций сложения. Итого, мы можем считать rank за время O(ℓ)O(\ell)O(ℓ) с предподсчётом за время O(n)O(n)O(n).

Для чисел массива LLL мы выделяем одинаковое количество бит, равное максимальному размеру числа — ⌊log⁡2n⌋+1\lfloor \log_2 n \rfloor + 1⌊log2​n⌋+1. В итоге мы храним всего (⌊log⁡2n⌋+1)⋅⌊n/ℓ⌋=O(nlog⁡n/ℓ)\bigl( \lfloor \log_2 n \rfloor + 1 \bigr) \cdot \lfloor n / \ell \rfloor = O(n \log n / \ell)(⌊log2​n⌋+1)⋅⌊n/ℓ⌋=O(nlogn/ℓ) бит.

Компромисса между скоростью и памятью достичь трудно. Мы хотим, чтобы rank считался как можно быстрее, а значит ℓ\ellℓ должно быть как можно меньше. А ещё мы хотим, чтобы памяти занималось как можно меньше, значит ℓ\ellℓ должно быть как можно больше. На самом деле компромисс искать не надо.

На минуту остановимся и подумаем, что вообще у нас получилось. Изначально мы пытались ответить на вопрос «какова сумма первых kkk битов?» Теперь, после разделения на блоки, мы пытаемся ответить на вопрос «какова сумма первых k mod ℓk \bmod \ellkmodℓ битов блока?» То есть мы свели исходную задачу к меньшей. А эту меньшую задачу можно решить уже отработанным методом — префиксные суммы.

Давайте для каждого блока с индексом s=⌊k/ℓ⌋s = \lfloor k / \ell \rfloors=⌊k/ℓ⌋ посчитаем префиксные суммы и сохраним их в массив RsR_sRs​. То есть

Rs[i]=∑j=sℓsℓ+ibjR_s [i] = \sum\limits_{j = s \ell}^{s \ell + i} b_j Rs​[i]=j=sℓ∑sℓ+i​bj​

И теперь, для вычисления операции rank, надо просто сложить два числа

rank⁡1(k)=L[⌊k/ℓ⌋]+R⌊k/ℓ⌋[k mod ℓ]\rank_1 (k) = L \bigl[ \lfloor k / \ell \rfloor \bigr] + R_{\lfloor k / \ell \rfloor} [k \bmod \ell]rank1​(k)=L[⌊k/ℓ⌋]+R⌊k/ℓ⌋​[kmodℓ]

А что по памяти? В массивах RsR_sRs​ мы храним в сумме nnn чисел, при этом каждое число не превосходит ℓ\ellℓ, ведь храним мы там количество бит в префиксе блока длины ℓ\ellℓ. Значит, каждое число занимает максимум ⌊log⁡2ℓ⌋+1\lfloor \log_2 \ell \rfloor + 1⌊log2​ℓ⌋+1 бит, а все массивы RsR_sRs​ занимают n⋅(⌊log⁡2ℓ⌋+1)n \cdot \bigl( \lfloor \log_2 \ell \rfloor + 1 \bigr)n⋅(⌊log2​ℓ⌋+1) бит. Итого вся дополнительная память — память на массив LLL и на все массивы RsR_sRs​:

(⌊log⁡2n⌋+1)⋅⌊n/ℓ⌋+n⋅(⌊log⁡2ℓ⌋+1)\bigl( \lfloor \log_2 n \rfloor + 1 \bigr) \cdot \lfloor n / \ell \rfloor + n \cdot \bigl( \lfloor \log_2 \ell \rfloor + 1 \bigr)(⌊log2​n⌋+1)⋅⌊n/ℓ⌋+n⋅(⌊log2​ℓ⌋+1)
